# TODO: Library 将来的な拡張課題

## 1. 優先度の高い拡張機能
### 1.1. 通知システム (Notification System)
- **概要**: 予約した本の準備が整った際や、返却期限が近づいた際にユーザーへ通知を届ける仕組み。
- **背景**: 現時点ではコア機能の完成を優先し、本機能の実装は将来課題として保留とする。
- **実装内容**:
    - 予約引き当て時の自動通知生成。
    - ヘッダーへの未読カウント表示。
    - 通知一覧画面の構築。

### 1.2. 外部API連携による書籍情報の自動取得
- **概要**: ISBNを入力またはスキャンするだけで、タイトル、著者、出版社、書影、あらすじを自動取得する機能。
- **対象API**: 楽天ブックスAPI、Google Books API等。
- **メリット**: 管理者の登録コストを劇的に削減し、データの正確性を向上させる。

### 1.3. 予約ロジックの高度化（在庫引き当て）
- **概要**: 在庫がある状態で予約が行われた際、その場で特定の「本（個体）」を確保（RESERVED状態へ変更）するロジック。
- **現状**: 現在は「誰かが借りている場合のみ予約可能」という暫定的なUIガードで対応中。

## 2. コミュニティ・UX拡張
### 2.1. レビュー・評価機能
- **内容**: 読んだ本に対して 5 段階評価とコメントを投稿できる機能。
- **目的**: 社内での知見共有をさらに活性化させる。

### 2.2. レコメンドエンジン
- **内容**: 貸出履歴に基づき、「この本を読んだ人は、こんな本も読んでいます」を表示する機能。

### 2.3. カート（まとめて貸出）機能
- **内容**: 複数の本を一時的にストックし、一括で貸出・予約手続きを行う機能。

## 3. 技術的負債・リファクタリング
### 3.1. 関連名 (related_name) の一掃と標準化
- **概要**: `catalog` アプリ内の各モデルに設定されている `related_name` を削除し、Django 標準の `_set` 形式へ統一する。
- **背景**: 大規模リファクタリング時の整合性確保を優先し、現在は暫定的に `related_name="books"` を維持している。
- **メリット**: マジックネームを排除し、Django の規約に忠実な透明性の高いコードベースを実現する。

### 3.2. 型チェックの高度化 (mypy/django-stubsの完全適合)
- **概要**: 現状抑制している Django 特有の動的な挙動や Mixin 構造に起因する型エラーを根本的に解決する。
- **実装内容**:
    - `django_stubs_ext.monkeypatch()` の導入による QuerySet/Manager のジェネリック化。
    - Mixin クラスへの `typing.Protocol` または `TYPE_CHECKING` を用いた属性（`self.request` 等）の明示。
    - Factory クラスの型定義改善（インスタンス属性と Factory クラス属性の混同解消）。
    - `from_queryset` を使用した Manager の明示的な型指定への移行。

### 3.3. アップロード画像ファイルの自動クリーンアップ
- **概要**: `Shelf` の配置写真や `Floor` のマップ画像が変更・物理削除された際に、サーバーのローカルストレージから不要になった画像ファイルを自動で削除する仕組みの導入。
- **背景**: 現時点ではポートフォリオとしてのコードの簡潔性と説明のしやすさを優先し、複雑になりやすい Django signals や外部パッケージへの依存を意図的に排除している。
- **解決策**: 将来的に運用規模が拡大した際は、`django-cleanup` ライブラリを導入するか、`pre_save` / `post_delete` シグナルを用いた画像クリーンアップハンドラを実装し、不要画像がストレージを圧迫する技術的負債を解消する。

## 4. フロントエンド・UX最適化

### 4.1. お気に入り登録の非同期化 (Ajax / htmx)
- **概要**: ページ全体のリロードを伴わず、ボタン部分のみを更新する非同期通信の実装。
- **メリット**: スムーズな操作感を実現し、モダンなWebアプリとしてのUXを向上させる。
- **手法**: `htmx` ライブラリを採用し、JavaScriptの記述を最小限に抑えつつ実現する。

## 5. 開発環境・品質管理の自動化

### 5.1. GitHub ActionsによるCIパイプラインの構築
- **概要**: GitHubへのプッシュおよびプルリクエスト時に、自動でDjangoテスト、Ruff、mypyを実行するCI環境の構築。
- **メリット**: 品質管理プロセスを自動化し、ポートフォリオとしてのプロフェッショナルな開発プロセスをアピールする。

### 5.2. pre-commitフックによるローカル検証の自動化
- **概要**: Gitコミット時に、ローカル環境で自動的にRuffやmypyを実行し、コード規格違反や構文エラーがある場合はコミットをブロックする仕組み。
- **メリット**: 手元での品質担保を徹底し、不要な修正コミットを削減する。
