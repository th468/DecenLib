# Project Instructions: Libarally

## 0. プロジェクトの背景と目的
- **ポートフォリオ:** このプロジェクトは就職活動のためのポートフォリオとして作成されています。
- **学習目的:** プログラミングスキルの向上、および開発ツールの効果的な活用方法を学習することを主眼に置いています。
- **設定:** 架空の会社からの依頼を受け、プロフェッショナルな品質で開発・納品するシミュレーションとして進行します。

## 1. プロジェクトの概要
- **名称:** Libarally
- **目的:** 蔵書の管理、ユーザーアカウント管理、および貸出（transactions）の管理を行うプロトタイプ。
- **主要な機能:**
  - `accounts`: カスタムユーザーモデル、ユーザー認証、部署（Department）管理。
  - `books`: 書籍情報（Biblio）、実体（Book）、および保管場所（Floor, Shelf）の管理。
  - `transactions`: 貸出（Lending）及び予約プロセスの管理。
  - `dashboard`: 管理者・利用者向けのダッシュボード。
  - `core`: プロジェクト全体で共有する基本クラス（BaseModel）やMixin。

## 2. 技術スタック
- **Framework:** Django 6.0.2
- **Language:** Python 3.14
- **Forms:** django-widget-tweaks
- **Testing:** Factory Boy, Faker (Django標準のTestCaseを使用)
- **Linter/Formatter:** Ruff (コード品質維持と自動整形の標準化)
- **Type Check:** mypy (型安全性の確保と実行前エラーの検知)
- **Database:** SQLite3 (開発用)

## 3. ディレクトリ構成
- `libarally/`: Django プロジェクトルート
  - `manage.py`: エントリポイント
  - `libarally/`: プロジェクト設定（settings.py, urls.py）
  - `accounts/`, `books/`, `transactions/`, `dashboard/`: 各機能アプリ
  - `core/`: 共通モデル・ミキシン
  - `templates/`: 全体共有テンプレート
    - `common/`: ベースレイアウトや共通パーツ
- `static/`: プロジェクト全体の共通静的ファイル（CSS, JS, Images）
- `media/`: ユーザーがアップロードするメディアファイル
- `docs/`: 設計ドキュメント、ER図、プロトタイプ資料、TODO.md 等
- `.env`: 環境変数（SECRET_KEYやDB設定等の機密情報の分離）

## 4. 開発・コーディング規約
### モデル
- `core.models.BaseModel` を可能な限り継承し、作成日時・更新日時の管理を一貫させる。
- **論理削除の標準化**: すべてのモデルは `core.models.BaseModel` を継承することで、`is_active` フラグによる論理削除を標準機能として利用すること。`delete()` メソッドの呼び出しは自動的に論理削除として処理される。

### ビューとロジックの分離
- **View:** CRUDなどの標準的な処理には **クラスベースビュー (CBV)** を優先し、コードの再利用性を高めること。複雑な遷移や特殊なバリデーションが必要な場合のみ **関数ベースビュー (FBV)** を検討すること。
- **Logic:** ビジネスロジックを View に直接書かないこと。以下の優先順位で配置する：
  1. **Model:** モデル単体に関する計算や状態変更（Fat Model）。
  2. **Form:** 複数フィールドにまたがる入力バリデーションやデータの加工。
  3. **Service (必要時):** 複数モデルを跨ぐ複雑なトランザクション処理。
- **Naming:** PEP8に準拠し、クラス名は `PascalCase`、変数・関数名は `snake_case` を徹底すること。

### テスト
- 各アプリの `tests/` ディレクトリ内にテストコードを配置する。
- テストデータの作成には `factories.py` で定義された Factory クラスを使用する。
- **共通基盤の活用:** `BaseModel` を継承したモデルのテストを行う際は、`core.tests.test_mixins.BaseModelTestMixin` を継承し、`factory_class` を指定することで、プロジェクト標準の機能（論理削除等）を自動検証すること。
- **__str__ メソッドのテスト:** モデルで `__str__` をオーバーライドした場合、必ずテストクラス側でも対応する検証メソッド（`run_str_test` 等）をオーバーライドし、独自の出力形式を厳密に検証すること。
- **網羅性と品質:**
  - **正常系:** Factory Boyを使用して「期待される最小限のデータ」でパスすることを確認する。
  - **異常系:** 不正な値の入力、権限のないユーザーによるアクセス、存在しないIDの指定など、エラーが適切にハンドリングされるかを確認する。
  - **境界値:** 最大文字数、最小値、日付の前後関係など、バグが起きやすい条件を意識したテストケースを作成すること。
  - **可読性:** テストメソッド名は `test_should_[期待される結果]_[条件]` の形式を推奨する。
- 新機能追加時は必ずテストコードを更新・追加すること。

### テンプレート
- `templates/common/base.html` をベースとし、`block` を適切に使用して拡張する。
- UI部品の共通化には `templates/common/includes/` を活用する。

## 5. 主要なコマンド
- **サーバー起動:** `python libarally/manage.py runserver`
- **テスト実行:** `python libarally/manage.py test libarally`
- **品質チェック (Ruff):** `ruff check .`
- **型チェック (mypy):** `mypy .`
- **マイグレーション作成:** `python libarally/manage.py makemigrations`
- **マイグレーション適用:** `python libarally/manage.py migrate`

## 6. コミュニケーションと言行
- **変更提案のプロセス:**
  - コードを実際に変更する前に、修正が必要な箇所を一覧化し、その理由（Rationale）を説明すること。
  - 局所的な修正よりも先に、構造上の問題（設計、データ構造、アーキテクチャ）がないかを確認し、問題がある場合はまずそれを報告すること。
- **代替案の提示:**
  - 構造上の問題を指摘する際は、必ず代替案（リファクタリング案）を提示すること。
  - 代替案は必ずしも具体的なコードである必要はなく、概念的な方向性の提示でもよい。
  - 可能であれば複数の案を提示すること。ただし、採用の可能性が著しく低いと思われる場合は、最も推奨される1案のみで構わない。

## 7. 学習と品質のための追加ルール
- **技術選定の解説:** 新しい手法やライブラリを提案する際は、学習目的のため、そのメリット・デメリットを比較・解説すること。
- **不足情報の確認:** コンテキストや要件が不足している、あるいは曖昧だと判断した場合は、推測で進めず、作業前に必ずユーザーに質問して解消すること。
- **TODO管理:** 実装時に発見した「将来的な改善点」や「技術的負債」は、別途 `TODO.md` に記録、またはコード内に明記すること。
- **コマンド実行の透明性:** 学習目的として、コマンドの内容と実行結果を一つずつ確実に把握するため、シェルコマンドを `&&` で連結せず、原則として個別に実行・確認すること。
- **重複コードの排除:** 新しいコードを追記または修正する際は、既存のコードと重複していないか、あるいは古いロジックが残っていないかを必ず確認し、不要なものは削除して常にクリーンな状態を保つこと。

## 8. 依頼主（架空）の要件詳細

### 8.1. プロジェクト背景と現状
- **現状:** オフィスの20フロアに約15,000冊の本が分散して存在しているが、有効活用されていない。
- **課題:** 図書館を作るスペースや管理コスト（人手）がない。
- **解決策:** ビル内に分散したまま「仮想の本棚」としてデータを集約し、スマホやPCから簡単に貸出・返却ができる「分散型図書館アプリ」を構築する。

### 8.2. 主要な機能要件
1.  **蔵書の登録（スマホ/PC対応）**:
    - 本のタイトル、著者、出版日、保管場所（フロア・詳細位置）等の情報を管理。
2.  **検索と閲覧**:
    - 分散した本を一括検索し、詳細画面で「貸出可」か「貸出中」を即座に判別可能にする。
3.  **貸出・予約プロセス**:
    - 貸出可の場合は「借りる」、貸出中の場合は「予約」ができるサイクル。
    - **利便性の工夫**: 貸出/予約確定時に、具体的な受け取り場所を提示する。

### 8.3. UI/UX・利便性に関する制約
- **場所情報の提示タイミング**: 貸出/予約のアクションを起こした際に初めて場所を表示する設計（興味を引くためのフロー）。
- **想定ユーザー**: 約4,000人の社員。ITに詳しくないユーザーでも直感的に操作できること。
